## 🏗️ 1. Architecture Concepts

### 🔹 Flow
* **Definition:** The root configuration container or structural blueprint of your data pipeline. It defines namespaces, core tasks, operational constraints, and execution orders.
* **OOP Analogy:** A Class definition.
* **YAML Placement:** Absolute root level. Must contain a unique `id` and a `namespace`.
* **Example:**
```yaml
id: market_data_ingestion
namespace: zoomcamp.production
```

### 🔹 Execution
* **Definition:** A single, live instance run of a specific flow. It contains its own isolated state machine, logs, metric tracking, and dynamic context.
* **OOP Analogy:** An Object instance instantiated from a Class.
* **YAML Placement:** Created dynamically at runtime. It cannot be hardcoded.
* **Execution States:**
  * 🟡 `RUNNING` — Tasks are currently active.
  * 🟠 `RETRYING` — A failed task is being retried per its retry policy.
  * 🟢 `SUCCESS` — All tasks executed successfully.
  * 🔴 `FAILED` — A task crashed, forcing the pipeline execution loop to terminate.
  * ⚪ `WARNING` — Completed, but one or more tasks reported a non-blocking issue.
  * ⏸️ `PAUSED` — Execution is waiting (e.g., on a `Pause` task or human approval).
  * ⚫ `KILLED` — Manually or forcibly terminated.

### 🔹 Tasks
* **Definition:** The fundamental modular blocks of execution within a flow. Tasks utilize specialized ecosystem plugins to interact with infrastructure, scripts, databases, and files.
* **OOP Analogy:** A Class method or function.
* **YAML Placement:** Declared under the root `tasks:` block as a collection array.
* **Example:**
```yaml
tasks:
  - id: extract_api_payload
    type: io.kestra.plugin.core.http.Request
    uri: "https://query1.finance.yahoo.com/v8/finance/chart/AAPL"
```

## ⚙️ 2. Dynamic Data & Context

### 🔹 Inputs
* **Definition:** Dynamic configurations or variables passed into a pipeline at the absolute beginning of runtime execution. Enforces strict strong typing.
* **OOP Analogy:** Function arguments or parameters.
* **YAML Placement:** Root level under `inputs:`.
* **Example:**
```yaml
inputs:
  - id: target_ticker
    type: STRING
    defaults: AAPL
    required: true
```
* **Usage in Tasks:** Referenced via `{{ inputs.input_id }}`.
```yaml
message: "Processing target ticker: {{ inputs.target_ticker }}"
```
* **Common input types:** `STRING`, `INT`, `FLOAT`, `BOOLEAN`, `DATETIME`, `DATE`, `URI`, `FILE`, `JSON`, `SELECT`, `MULTISELECT`, `SECRET`.

### 🔹 Outputs
* **Definition:** Structured data matrices, paths, or response payloads generated by a finished task and sent downstream to subsequent tasks.
* **OOP Analogy:** Function return values.
* **YAML Placement:** Inherited by referencing prior task IDs using the prefix `outputs.`.
* **Example:**
```yaml
tasks:
  - id: fetch_weather
    type: io.kestra.plugin.core.http.Request
    uri: "https://api.open-meteo.com/v1/forecast?latitude=41.87&longitude=-87.62&daily=temperature_2m_max"

  - id: log_weather_response
    type: io.kestra.plugin.core.log.Log
    message: "API Body: {{ outputs.fetch_weather.body }}"
```
* **Flow-level outputs:** A flow itself can expose outputs (declared under root `outputs:`), letting subflows return values to a parent flow just like a function returns a value to its caller.

### 🔹 Variables
* **Definition:** Internal key–value pairs defined at the flow level for code cleanup and global reuse within that specific blueprint file.
* **OOP Analogy:** Local constants or variables.
* **YAML Placement:** Declared at the root level under `variables:`.
* **Example:**
```yaml
variables:
  base_data_path: "/tmp/storage/market_dw"
  nested_msg: "Ingesting to path: {{ vars.base_data_path }}"

tasks:
  - id: print_path
    type: io.kestra.plugin.core.log.Log
    message: "{{ render(vars.nested_msg) }}"
```
> 💡 **Tip:** If a variable references another expression (like `inputs` or `vars`), you must wrap its consumption in the `{{ render(vars.variable_id) }}` function to execute deep parsing.

### 🔹 Built-in System Variables
Kestra automatically exposes context variables you can reference anywhere in a flow:

| Variable | Description |
|---|---|
| `{{ flow.id }}` | ID of the current flow |
| `{{ flow.namespace }}` | Namespace of the current flow |
| `{{ execution.id }}` | Unique ID of the current execution |
| `{{ execution.startDate }}` | Timestamp the execution began |
| `{{ task.id }}` | ID of the currently running task |
| `{{ taskrun.startDate }}` | Timestamp the current task started |

## 🔀 3. Flow Control Tasks

Kestra treats control-flow constructs as **flowable tasks** — tasks that orchestrate other tasks rather than execute business logic themselves.

### 🔹 If / Else
* **Definition:** Conditional branching based on an expression.
```yaml
- id: check_ticker
  type: io.kestra.plugin.core.flow.If
  condition: "{{ inputs.target_ticker == 'AAPL' }}"
  then:
    - id: log_apple
      type: io.kestra.plugin.core.log.Log
      message: "Tracking Apple stock"
  else:
    - id: log_other
      type: io.kestra.plugin.core.log.Log
      message: "Tracking a different ticker"
```

### 🔹 Switch
* **Definition:** Multi-branch routing analogous to a `switch`/`case` statement.
```yaml
- id: route_by_ticker
  type: io.kestra.plugin.core.flow.Switch
  value: "{{ inputs.target_ticker }}"
  cases:
    AAPL:
      - id: apple_branch
        type: io.kestra.plugin.core.log.Log
        message: "Apple pipeline"
    SPY:
      - id: spy_branch
        type: io.kestra.plugin.core.log.Log
        message: "SPY pipeline"
  defaults:
    - id: default_branch
      type: io.kestra.plugin.core.log.Log
      message: "Generic pipeline"
```

### 🔹 ForEach / EachParallel
* **Definition:** Iterates over a list, running a task subtree per item — sequentially (`ForEach`) or concurrently (`EachParallel`).
```yaml
- id: process_tickers
  type: io.kestra.plugin.core.flow.ForEach
  values: ["AAPL", "MSFT", "GOOG"]
  tasks:
    - id: log_ticker
      type: io.kestra.plugin.core.log.Log
      message: "Processing {{ taskrun.value }}"
```

### 🔹 Parallel
* **Definition:** Runs a set of independent tasks concurrently instead of sequentially.
```yaml
- id: parallel_fetch
  type: io.kestra.plugin.core.flow.Parallel
  tasks:
    - id: fetch_a
      type: io.kestra.plugin.core.http.Request
      uri: "https://api.example.com/a"
    - id: fetch_b
      type: io.kestra.plugin.core.http.Request
      uri: "https://api.example.com/b"
```

### 🔹 Subflow (Flow Task)
* **Definition:** Calls another flow as a reusable, isolated unit — enabling modular, composable pipelines.
* **OOP Analogy:** A function call to another module.
```yaml
- id: trigger_child_pipeline
  type: io.kestra.plugin.core.flow.Subflow
  namespace: zoomcamp.production
  flowId: market_data_ingestion
  inputs:
    target_ticker: MSFT
  wait: true
  transmitFailed: true
```

## 🛡️ 4. Error Handling & Resilience

### 🔹 Retries
* **Definition:** Automatically re-attempts a failed task before marking the execution as failed.
```yaml
- id: fetch_weather
  type: io.kestra.plugin.core.http.Request
  uri: "https://api.open-meteo.com/v1/forecast"
  retry:
    type: constant
    interval: PT10S
    maxAttempt: 3
```
* **Retry types:** `constant`, `exponential`, `random`.

### 🔹 Timeout
* **Definition:** Caps how long a task is allowed to run before Kestra forcibly kills it.
```yaml
- id: long_running_query
  type: io.kestra.plugin.jdbc.postgresql.Query
  timeout: PT5M
```

### 🔹 allowFailure & allowWarning
* **Definition:** Lets a task fail or warn without halting the entire execution — useful for optional/non-critical steps.
```yaml
- id: optional_notification
  type: io.kestra.plugin.notifications.slack.SlackIncomingWebhook
  allowFailure: true
```

### 🔹 Errors Block (Flow-level Error Handling)
* **Definition:** A dedicated set of tasks that run only if the main flow fails — ideal for alerting, cleanup, or rollback logic.
* **OOP Analogy:** A `catch` block.
```yaml
errors:
  - id: alert_on_failure
    type: io.kestra.plugin.notifications.slack.SlackIncomingWebhook
    url: "{{ secret('SLACK_WEBHOOK') }}"
    payload: "Flow {{ flow.id }} failed on execution {{ execution.id }}"
```

### 🔹 finally Block
* **Definition:** Runs regardless of success or failure — analogous to a `finally` clause — commonly used for teardown (e.g., closing connections, removing temp files).
```yaml
finally:
  - id: cleanup_temp_files
    type: io.kestra.plugin.core.log.Log
    message: "Cleaning up after execution {{ execution.id }}"
```

## 🛠️ 5. Operations & Safety Controls

### 🔹 Triggers
* **Definition:** Automatic triggers designed to launch new flow executions based on schedules or external infrastructure events.
* **OOP Analogy:** Event Listeners, Alarms, or Crontab hooks.
* **YAML Placement:** Root level under `triggers:`.
```yaml
triggers:
  - id: daily_market_close
    type: io.kestra.plugin.core.trigger.Schedule
    cron: "0 17 * * 1-5"  # Mon-Fri at 5:00 PM
    inputs:
      target_ticker: SPY
```

### 🔹 Plugin Defaults
* **Definition:** Flow-level overrides that inject uniform key values into every single instance of a specific plugin type automatically.
* **OOP Analogy:** Configuration base inheritance.
* **YAML Placement:** Root level under `pluginDefaults:`.
```yaml
pluginDefaults:
  - type: io.kestra.plugin.core.log.Log
    values:
      level: ERROR  # Automatically forces all Log tasks to log as ERROR
```

### 🔹 Concurrency
* **Definition:** Defensive flow-control blocks that limit and restrict the number of identical flow executions running in parallel on your host clusters.
* **OOP Analogy:** Thread limiters or Semaphores.
* **YAML Placement:** Root level under `concurrency:`.
```yaml
concurrency:
  limit: 1
  behavior: FAIL  # Aborts any scheduled/manual runs if an instance is already active
```
* **Behavior options:** `FAIL`, `QUEUE`, `CANCEL`.

### 🔹 Working Directory
* **Definition:** A shared, isolated file-system scratch space for a group of tasks that need to pass files (not just structured outputs) between each other.
```yaml
- id: wdir
  type: io.kestra.plugin.core.flow.WorkingDirectory
  tasks:
    - id: download_file
      type: io.kestra.plugin.core.http.Download
      uri: "https://example.com/data.csv"
    - id: process_file
      type: io.kestra.plugin.scripts.python.Script
      script: |
        import pandas as pd
        df = pd.read_csv("data.csv")
        print(df.describe())
```

## ⏱️ 6. Triggers Deep Dive

| Trigger Type | Fires When | Typical Use Case |
|---|---|---|
| `Schedule` | A cron expression matches | Daily/weekly batch jobs |
| `Webhook` | An HTTP POST hits a unique URL | External systems / CI/CD callbacks |
| `Flow` | Another flow reaches a certain state | Chaining pipelines together |
| `io.kestra.plugin.fs.*` (e.g., S3/SFTP triggers) | A file lands in a monitored location | Event-driven ingestion |

```yaml
triggers:
  - id: on_upstream_success
    type: io.kestra.plugin.core.trigger.Flow
    conditions:
      - type: io.kestra.plugin.core.condition.ExecutionStatusCondition
        in: [SUCCESS]
    preconditions:
      flows:
        - namespace: zoomcamp.production
          flowId: market_data_ingestion
```

## 🧩 7. Organization & Reuse

### 🔹 Namespaces
* **Definition:** Logical grouping/segmentation of flows (e.g., by team, environment, or domain) — also scopes namespace-level variables and secrets.

### 🔹 Labels
* **Definition:** Free-form key-value metadata tags attached to executions for filtering, cost-tracking, and search in the UI.
```yaml
labels:
  team: data-eng
  env: production
```

### 🔹 Namespace Files
* **Definition:** Shared, version-controlled files (scripts, SQL, configs) accessible to any flow within the same namespace — avoids duplicating logic across flows.

## 👁️ 8. Observability

* **Logs:** Each task run streams structured logs, filterable by level (`TRACE`→`ERROR`).
* **Metrics:** Kestra emits execution duration, task duration, and custom metrics that can be scraped by Prometheus/Grafana.
* **Gantt & Topology Views:** The UI visualizes execution timing (Gantt) and task dependency graphs (Topology) for debugging.
* **Backfill:** Re-run a `Schedule` trigger across a historical date range to fill in missed or corrected executions.

## 🎛️ 9. Master Blueprint Layout Reference

```yaml
id: 01_hello_world
namespace: zoomcamp

labels:
  team: data-eng
  env: dev

inputs:
  - id: name
    type: STRING
    defaults: Will

concurrency:
  behavior: FAIL
  limit: 2

variables:
  welcome_message: "Hello, {{ inputs.name }}!"

tasks:
  - id: hello_message
    type: io.kestra.plugin.core.log.Log
    message: "{{ render(vars.welcome_message) }}"

  - id: generate_output
    type: io.kestra.plugin.core.debug.Return
    format: I was generated during this workflow.

  - id: sleep
    type: io.kestra.plugin.core.flow.Sleep
    duration: PT15S

  - id: log_output
    type: io.kestra.plugin.core.log.Log
    message: "This is an output: {{ outputs.generate_output.value }}"

  - id: goodbye_message
    type: io.kestra.plugin.core.log.Log
    message: "Goodbye, {{ inputs.name }}!"
    retry:
      type: constant
      interval: PT5S
      maxAttempt: 2

errors:
  - id: notify_failure
    type: io.kestra.plugin.core.log.Log
    message: "Execution {{ execution.id }} of flow {{ flow.id }} failed."

finally:
  - id: cleanup
    type: io.kestra.plugin.core.log.Log
    message: "Execution finished — running cleanup."

pluginDefaults:
  - type: io.kestra.plugin.core.log.Log
    values:
      level: ERROR

triggers:
  - id: schedule
    type: io.kestra.plugin.core.trigger.Schedule
    cron: "0 10 * * *"
    inputs:
      name: Sarah
    disabled: true
```

## 📖 10. Glossary Quick-Reference Table

| Kestra Concept | OOP Analogy | Root YAML Key |
|---|---|---|
| Flow | Class | *(top-level file)* |
| Execution | Object instance | *(runtime only)* |
| Task | Method / function | `tasks:` |
| Input | Function argument | `inputs:` |
| Output | Return value | `outputs.<task_id>` |
| Variable | Local constant | `variables:` |
| Trigger | Event listener / cron | `triggers:` |
| Subflow | Function call to another module | `type: ...flow.Subflow` |
| Errors block | `catch` block | `errors:` |
| Finally block | `finally` block | `finally:` |
| Plugin Defaults | Base config inheritance | `pluginDefaults:` |
| Concurrency | Thread limiter / semaphore | `concurrency:` |
| Working Directory | Shared temp filesystem | `type: ...flow.WorkingDirectory` |
| Labels | Metadata tags | `labels:` |

### ✅ Best-Practice Checklist
- [ ] Every production flow has an `errors:` block for alerting.
- [ ] Long-running tasks have explicit `timeout` values.
- [ ] Flaky I/O tasks (HTTP, DB) have a `retry` policy.
- [ ] Non-critical steps use `allowFailure: true` instead of blocking the whole run.
- [ ] Related flows share a consistent `namespace` convention.
- [ ] Executions are tagged with `labels` for cost/ownership tracking.
- [ ] Concurrency limits are set on flows triggered by schedules to avoid overlapping runs.
